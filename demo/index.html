Synthetic fraud simulation CLI

Build safer fraud systems without touching real customer data.

fintech-fraud-sim is a TypeScript command-line tool that generates realistic synthetic fintech users, accounts, devices, beneficiaries, merchants, transactions, events, risk scores, fraud patterns, reports, and exports — for QA, fraud operations, AML demos, dashboards, and model prototypes.

What it does

Creates realistic fraud scenarios

Model mule accounts, account takeover, velocity abuse, fraud rings, KYC abuse, marketplace refunds, chargebacks, structuring, layering, and cross-border anomalies.

Why it matters

Reduces risky test data habits

Teams test dashboards, review queues, rule engines, APIs, reports, and ML pipelines without copying production records or exposing sensitive identifiers.

Impact

Speeds up fraud product work

Product, data, compliance, QA, and engineering teams get repeatable fixtures that make demos clearer, regressions easier, and risk workflows faster to validate.

!

Synthetic-only safety promise

All generated data is synthetic. It contains no real names, phone numbers, emails, BVNs, NINs, bank account numbers, or any real personal data. Identifiers are fabricated and labels like bvn_like_id or synthetic_bvn_check are intentionally fictional. Use it for testing, education, and fraud-model prototyping only — never as a stand-in for production data or a real watchlist.

Quick start

Up and running in three steps

Requires Node.js 22 or newer. No build step or database is needed — everything runs locally.

1

Run with npx

No install required — run the latest version directly.

npx fintech-fraud-sim --help
2

Preview a sample

Print a small sample to your terminal without writing files.

npx fintech-fraud-sim preview \
  --users 20 --fraud-rate 0.15 \
  --limit 5 --pretty
3

Generate a dataset

Write full CSV + JSON fixtures to an output directory.

npx fintech-fraud-sim generate \
  --users 1000 --fraud-rate 0.08 \
  --seed demo --out ./data

Local development

Clone the repo and build from source if you want to extend it.

npm install
npm run build
npm test
npm run dev -- generate --users 100 --fraud-rate 0.1 --seed demo

Fraud patterns

17 built-in typologies

Each pattern injects characteristic signals and reason codes so detection systems have something realistic to catch. Combine them with --patterns and tune the mix with --pattern-weights.

mule_account

New account, high beneficiary count, frequent funds movement.

account_takeover

Device change, IP country mismatch, failed login spike.

velocity_abuse

Many transactions in a short period, often below thresholds.

kyc_abuse

Failed KYC attempts, reused device, inconsistent country data.

chargeback_risk

Prior chargebacks and high-value transactions.

transaction_spike

Transaction amount far above the user's baseline.

cross_border_anomaly

Declared country differs from IP or beneficiary country.

beneficiary_burst

Many new beneficiaries added within 24 hours.

fraud_ring

Coordinated accounts sharing network_id, devices, beneficiaries.

synthetic_identity

New identity with repeated KYC friction and inconsistent signals.

friendly_fraud

Customer-like behavior that later produces disputes/reversals.

promo_abuse

Low-value high-frequency activity exploiting incentives.

merchant_collusion

Suspicious users and high-risk merchants linked by routed payments.

refund_abuse

Refund-heavy behavior with repeat reversals.

sanctions_false_positive

Synthetic watchlist-like review signals — no real sanctions.

structuring

Repeated transactions intentionally kept below review thresholds.

layering

Rapid movement across rails/beneficiaries to mimic AML layering.

Risk scoring and recommended actions

Every user and transaction carries a risk_score from 0–100 mapped to a recommended action, so you can test allow / review / block flows end to end.

allow · 0–44

Low-risk records for normal test traffic.

review · 45–74

Queues for manual review, rules testing, dashboard triage.

block · 75–100

High-risk simulations for decline/block flows.

Outputs & data model

Seven linked entity types, many formats.

Datasets are relational: users own accounts and devices, add beneficiaries, and transact with merchants. Shared network_id, device_id, and beneficiary_id values let graph systems reconstruct fraud rings.

Entities generated

  • users — KYC status, device count, risk score, fraud pattern, reason codes.
  • accounts — type, currency, balance, status, daily limit, fraud link.
  • devices — type, OS, first/last seen, trust flag, network link.
  • beneficiaries — country, bank code, recency, fraud link.
  • merchants — category, country, risk tier, high-risk flag.
  • transactions — amount, rail, channel, status, risk score, action.
  • events — derived stream of KYC, device, beneficiary, and decision events.

Output formats

CSV JSON NDJSON SQL inserts Parquet (fallback)

Use --format both for CSV + JSON, or --format all to write every format at once. A summary.json with dataset totals and fraud breakdowns is always included.

Note: Parquet uses a dependency-free fallback writer — .parquet files contain columnar JSON plus a manifest. Swap in a binary Parquet integration before loading into strict Parquet-only tools.

Programmatic API

Import it as a TypeScript library.

Generate and validate datasets in code — fully typed, deterministic with a seed, and ready to wire into test suites or pipelines.

import { generateDataset, validateDataset } from 'fintech-fraud-sim';

const dataset = generateDataset({
  users: 1000,
  fraudRate: 0.08,
  format: 'both',
  out: './output',
  country: 'NG',
  patterns: ['mule_account', 'account_takeover', 'structuring'],
  seed: 'demo',
  useCase: 'consumer_fintech'
});

const validation = validateDataset(dataset);
if (!validation.valid) console.error(validation.issues);

Library users can also register custom country profiles and platform presets via defineGenerationPlugin and registerGenerationPlugin.

FAQ

Common questions

Is any of this data real?+

No. Every record is synthetic. There are no real names, emails, phone numbers, government IDs, or account numbers. It is safe to commit fixtures to a repo or share in a demo.

Will I get the same data every run?+

Yes, if you pass a --seed. The same seed and options always produce byte-identical datasets, which makes regression tests and benchmarks reproducible.

What do I need to install?+

Just Node.js 22 or newer. Run it with npx fintech-fraud-sim — no global install required. Runtime dependencies are limited to @faker-js/faker and commander.

Can I use it for machine learning?+

Yes. ml-export writes train/test/validation feature matrices and labels, with optional stratified splits and leakage-field controls. Pair it with evaluate to score model predictions against ground truth.

Does it work outside Nigeria?+

Yes. Country profiles cover NG, US, GB, EU, KE, GH, and ZA, and platform presets and cross-border benchmark suites support UK and global corridors. You can also register your own profiles.

Where do I report bugs or request features?+

Open an issue on the GitHub issue tracker. The project is MIT licensed and contributions are welcome.
